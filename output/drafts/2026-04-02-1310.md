Shipped the biggest refactor of InsightClips this month — and it almost broke everything.

For those who don't know: InsightClips is an AI app that turns long-form YouTube videos into digestible bite-sized clips with key insights extracted.

The problem? I was polling AssemblyAI every few seconds to check if transcriptions were done. 

Imagine refreshing your email every 5 seconds waiting for that one important message. Except I was doing this for every single video upload.

It was:
- Slow (users waited 5+ minutes for results)
- Expensive (burning API calls like crazy) 
- Unreliable (timeouts everywhere)

The fix seemed simple: switch to webhooks. AssemblyAI calls me when transcription is ready, instead of me constantly asking "Are we there yet?"

Here's what I didn't expect:

1. **Database states are everything**
I had to track 4 different processing states: uploaded → transcribing → extracting → complete. 
One missing state transition and videos disappeared into the void.

2. **Webhooks fail silently** 
Unlike polling where you know immediately if something's wrong, webhooks just... don't arrive sometimes.
Had to build retry logic and fallback polling as backup.

3. **Testing async flows is brutal**
You can't just run the function and see what happens. I spent hours debugging what turned out to be a webhook that was hitting the wrong endpoint (classic me).

But here's what made it worth it:
- Processing time dropped from 5+ minutes to under 2 minutes
- API costs reduced by 60%
- Users actually get real-time status updates now

The biggest learning? Sometimes the "simple" architectural change is the hardest one to implement. But when you ship it right, everything else becomes easier.

Now I'm bumping video limits from 90 to 120 minutes because the system can actually handle it.

What's the gnarliest refactor you've had to ship recently?

Built a video transcription pipeline that actually works.

Most people think switching from polling to webhooks is just a minor optimization.
It's not.

When I first built InsightClips (extracts key moments from long videos), I used the lazy approach: poll AssemblyAI every few seconds to check if transcription was done.

Worked fine for demos.
Completely fell apart at scale.

The problem wasn't just API rate limits — though those hurt.
The real issue was user experience.

People would upload a 2-hour video and... wait. And wait. With zero feedback.
Some thought the app was broken and left.

So I spent the last few weeks ripping out the polling system and rebuilding it with webhooks.

Here's what I learned (the hard way):

**Webhooks aren't just "better polling"**
They're a completely different mental model.
Instead of "ask until you get an answer," it's "tell me when you're done."

This meant rebuilding the entire job tracking system.
And handling edge cases I never considered with polling.

**Infrastructure complexity explodes**
Polling lives in your app. Webhooks live... everywhere.
Had to set up proper webhook validation, replay mechanisms, and dead letter queues.

What used to be 50 lines of "check every 30 seconds" became 200+ lines of webhook handling.

**But the UX transformation was worth it**
Users now get real-time progress updates.
Videos process in the background.
No more mysterious waiting screens.

The technical debt? Completely justified.

Sometimes the "right way" is 4x more code.
But if it makes the product actually usable — ship it.
